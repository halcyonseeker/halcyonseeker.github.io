A New Internet Communications Protocol and Service

TL;DR

A proposal for a federated, open communications platform as a 21st century successor to IRC and as a competitor to Skype, Slack, and Discord.

I am an amateur. I got into tech about two years ago when I started a CS class in high school. I'm mostly self-taught. There are huge gaps in my knowledge. Rational

This document outlines the specifications for a new Internet communications protocol and an accompanying service intended as a FOSS (Free and Open Source Software) replacement for existing communication services like Skype, Discord, Slack, and Whatsapp, and as an extension of the IRC (Internet Relay Chat) protocol into the multimedia era and the contemporary mainstream. In the current year, the majority of mainstream communication occurs over non-free, non-private, and centralized services; with privacy and freedom respecting alternatives that work well being few and far between. Existing FOSS alternatives include:

Email Though email (if paired with good asymmetric encryption) is perhaps the ideal method for safe and secure communications and low-bandwidth file sharing, it has a few major downsides: Firstly, email is asynchronous; it is not a live form of communication and as such is not well suited to seamless conversation and collaboration. Secondly, though it was originally conceived as a federated communications system, in the present era email is all but monopolized by an advertising corporation (Google), with FOSS and privacy-respecting alternatives providers being rare, and an email server being impractically difficult to set up and maintain.

Signal The Signal messaging app is based on free software and makes use of strong encryption schemes, but it is centralized and requires the use of a smartphone. Any service that requires a smartphone as a crucial node is inherently insecure given the extremely proprietary and black-box nature of modern smartphones.

IRC Internet Relay Chat allows for user-controlled communication, however given its relatively obscure nature it and lack of support for VoIP and video communication, it would not effectively fill the role of an ideal, real-time communications system. It is worth noting that the functionality of existing services like Slack and Discord are heavily based on IRC, and the system proposed here will draw heavily from that protocol as well.

While they all excel to varying degrees and in their various roles as privacy-protecting communication media, all fall short in one or more of the design criteria for the new protocol.

Overview

Design Criteria

  • 100% powered by free and open-source software, ideally licensed under the GPL. No reliance on proprietary software. Caveat: client software may be built for proprietary systems like MS Windows or Apple's iOS.
  • Easy for non-tech savvy individuals to take advantage of; there's no point in having a cool communication platform if nobody uses it (this is the rational behind the ability for servers to be linked into and controlled by a network). IRC's main disadvantage is its lack of accessibility to the layperson.
  • Partially decentralized; though it does follow a client-server model, this protocol should not be reliant on the existence of a single company.
  • Client software should be made for as wide a range of platforms as possible in order to maximize availability and user choice.
  • Encrypted communications. All communications must be encrypted between client and server software, with the ability given to encrypt data stored on the server, optionally using users' existing PGP keys.
  • All client and server software would preferably make minimal use of system resources, no Electron apps, no Java.

Basic Concepts

Servers

This protocol will follow a client-server model where client programs connect to central servers in order to exchange information with other clients. Each server can contain multiple channels, be incorporated into networks, and be controlled by administrator client accounts. Server software can be hosted by anyone, or centralized into a network.

Clients

Clients are the software used to connect to the server. They can be web, command-line, or standalone-graphical applications. For the sake of cohesion, networks might want to maintain an official client.

Networks

Networks are groupings of servers under a single domain, in a similar manner to Discord. Networks allow for easier and more cohesive access to servers, allowing multiple server identities to be controlled centrally by network login credentials. Networks would also provide users within the network with the ability to more easily talk one-on-one with each other. Server admins should have the ability to immigrate and emigrate to and from networks, with the option to pull the their server and all its logs from a network and host it independently.

Channels

Initially, each server would have three channels, one for text/file sharing, one for voice, and one for video chat. A theoretically infinite set of channels could be added in each category. Channels would be designated by a hash and possess a description and permissions. A channel's permissions, set by a server's admin would dictate who can view, post to, and edit the attributes of the channel.

Direct Messaging

Direct text, voice, and video interaction can take place between two members of a same server, with the server serving as the mediator between the two clients. In the event that two people without a shared server wish to communicate directly, they would have to both be connected to a network of servers, where the data exchanged between the users would be associated with their accounts on the network instead of the shared server.

Messages and File Sharing

The text channels may also be used to share files including documents and multimedia. Maximum sizes would be determined by server or network administrators. The manner in which files are treated when opened could vary by client.

Encryption and Security

Voice, video, and text/file-sharing channels should be encrypted between clients and servers. Further encryption of channels and direct messages, as well as of content on the server can be achieved via PGP keys. For those using a network or running a server on a machine beyond their physical control, it is recommended to encrypt the content server-side. Owing to the relatively steep learning curve of public-private key cryptography and the broad target audience of this protocol, it may be wise to create some easier way of managing encryption keys.

Gateways

Servers and networks may implement gateways that would allow access over non-conventional networks, like cellular/sms, and uhf/vhf packet radio. Such gateways may be used let people place long-distance phone calls free of charge, or assist in disaster response and coordination efforts when conventional communications may be unreliable or down.

An Example Service

This is an example of a communications platform based on this protocol, with users able to register servers on hardware controlled by the platform and to access them with an official client for desktop and mobile form factors and the web. A new user would sign up with the platform, and thereby register with its network, choosing a username. Their account (information stored on one of the network's central administration servers) would be a standalone system, with an associated email being optional. Once on the network, the user would be able to search out other users on the network, open direct channels with them, and save them in their network contacts. This platform may choose to refer to a person's contacts as "friends". If they desire to communicate with multiple people, perhaps a group of friends, family, or colleagues, they can join or create a server. Though they may set custom nicknames, profile pics, and descriptions in each server they join, the users' identities are all collated around their network username. Should the user choose to host their own server they would have the option to connect it with the network they signed up with, thereby letting users of the network easily find an access it.
